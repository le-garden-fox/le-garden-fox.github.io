The Boot Sequence

Why you should know about this?

Simply put, you should know about the boot process because it introduces you to the fundamental pieces of your computing system. Having a working knowledge of the kernel, the init process, the BIOS and the other underlying process can give you the upper hand in debugging your system.

What are the steps in the boot process?

  • Power
  • BIOS
  • MBR
  • GRUB
  • Kernel
  • Init
  • Run-Levels or PBMGKIR or (Please Bring More Green Kellogs In Rice)

Power

  • This is self explanatory and is the manual step of allowing power to run through the system

BIOS

  • The BIOS or Basic Input Output System is the initial software that runs on your motherboard is dictated by the manufacturer.
  • The main purpose of the BIOS is to find the MDR and load it
  • In the BIOS you can define some general parameters about the system like
    • Where to find the MBR
    • Hardware options like overclocking
    • Password protect BIOS

MBR

  • The goal of the MBR, which is found in the first sector of the boot disk, is to load GRUB or whatever boot loader is configured
  • Contains a table of all the partitions on the disk that have been formatted

GRUB

  • It loads the kernel
  • Can be configured dynamically
  • Can be found on any device, i.e. disk, USB, CD, DVD
  • Can load OS even from network
  • Does not need to be rebuilt each time a config is changed

Kernel

  • It is the interface for the operating system that grants access to the hardware
  • Manages user space processes by scheduling it’s CPU time
  • Manages memory and creates the interface for virtual memory
  • Allocates memory for itself that the user space memory cannot access
  • Launches the init process

Init

  • Is the initial process that all user space fork from

Run Levels

  • Are the states of the init or system
  • Defines what systems are operational
    • 0 = Power-off
    • 1 = Single user mode (minimal functionality maintenance mode)
    • 2 = Multi user mode (No network interfaces or daemons)
    • 3 = Multi user mode with networking (basic run level for command line)
    • 4 = NA
    • 5 = X11 or GUI (GUI operating systems ready)
    • 6 = Reboot (reboots the system, starts shutting off services in order)

Systemd runlevels

  • Systemctl isolate <multi-user.target>
  • Run level 0 is matched by power-off.target (and runlevel0.target is a symbolic link to power-off.target).
  • Run level 1 is matched by rescue.target (and runlevel1.target is a symbolic link to rescue.target).
  • Run level 3 is emulated by multi-user.target (and runlevel3.target is a symbolic link to multi-user.target).
  • Run level 5 is emulated by graphical.target (and runlevel5.target is a symbolic link to graphical.target).
  • Run level 6 is emulated by reboot.target (and runlevel6.target is a symbolic link to reboot.target).
  • Emergency is matched by emergency.target.

Commands

  • Init will change run levels in older systems
  • Who -r (will display the current run level)
  • dmesg (shows the logs from the kernel) or driver message command will output the kernel ring buffer logs
  • /var/log/boot.log (system boot logs)
  • /var/log/dmesg (kernel logs)
  • Change run level with systemctl get-default
